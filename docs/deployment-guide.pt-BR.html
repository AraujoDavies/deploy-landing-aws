1Pré-requisitos

Antes de executar qualquer comando Terraform, confira se as ferramentas e os acessos abaixo estão prontos.

Ferramentas necessárias

  • Terraform ≥ 1.6 — terraform -version
  • AWS CLI v2 — aws --version
  • Credenciais AWS configuradas — aws configure ou variáveis de ambiente

Permissões AWS necessárias

O usuário ou role IAM que executa o Terraform precisa de permissões para: S3, CloudFront, ACM (us-east-1), Route 53 e KMS (se for reativado).

Bucket S3 do remote state (backend.tf)

Antes do primeiro terraform init, edite o backend.tf e substitua os placeholders:

terraform {
  backend "s3" {
    bucket  = "YOUR-TFSTATE-BUCKET-NAME"   # deve existir previamente
    key     = "static-website/terraform.tfstate"
    region  = "us-east-1"
    encrypt = true
  }
}
⚠ Importante
Crie o bucket S3 de state manualmente (ou com um script de bootstrap) antes de rodar terraform init. O Terraform não consegue criar o próprio backend. Lock de state (DynamoDB) não é usado — evite executar terraform apply em paralelo a partir de várias máquinas.

3Deploy em Dev

O ambiente de desenvolvimento usa o certificado padrão *.cloudfront.net — não é necessário domínio personalizado. É a forma mais rápida de colocar o site no ar.

Passo 1 — Inicializar o Terraform

terraform init

Baixa o provider AWS e configura o backend S3. Execute uma vez por máquina.

Passo 2 — Plan & Apply

# Prévia do que será criado (sem alterações)
terraform plan -var-file="envs/dev.tfvars"

# Cria todos os recursos
terraform apply -var-file="envs/dev.tfvars"

O Terraform criará: 1 bucket S3 (conteúdo), 1 distribuição CloudFront. Leva cerca de 3 minutos.

Passo 3 — Enviar arquivos HTML

Bash / macOS / Linux

# Obtém o nome do bucket a partir dos outputs
BUCKET=$(terraform output -raw content_bucket_name)

# Envia as duas páginas
aws s3 cp index.html s3://$BUCKET/index.html
aws s3 cp error.html s3://$BUCKET/error.html

PowerShell (Windows)

# Obtém o nome do bucket a partir dos outputs
$BUCKET = terraform output -raw content_bucket_name

# Envia as duas páginas
aws s3 cp index.html "s3://$BUCKET/index.html"
aws s3 cp error.html "s3://$BUCKET/error.html"

Passo 4 — Abrir o site

terraform output website_url
# → https://d1234abcd.cloudfront.net
✓ Dev no ar
O site está acessível pela URL do CloudFront. Nenhuma alteração no registrador de domínio é necessária.

4Deploy em Prod (Domínio Personalizado)

Pré-requisitos do domínio

  1. Tenha um domínio próprio Compre/registre um domínio em qualquer registrador (GoDaddy, Namecheap, Registro.br, etc.) ou no AWS Route 53 Registrar. O Terraform não compra o domínio — ele só cria a hosted zone de DNS.
  2. Defina domain_name no prod.tfvars O Terraform criará a hosted zone pública no Route 53 automaticamente no apply.
  3. Copie os nameservers dos outputs do Terraform Depois que a zone for criada, execute terraform output nameserver_setup_instructions (ou route53_nameservers).
  4. Atualize os nameservers no registrador Veja a Seção 5 para o passo a passo por registrador. A validação do ACM só conclui depois que essa alteração propagar.

Configurar prod.tfvars

Abra envs/prod.tfvars e substitua o placeholder do domínio:

domain_name = "ataqtech.com.br"   # ← domínio que você já possui

Plan & Apply (duas fases recomendadas)

O ACM precisa que o registrador use os nameservers do Route 53 antes de concluir a validação. Crie a hosted zone primeiro, atualize o registrador e só então finalize o stack:

Fase 1 — criar apenas a hosted zone

terraform apply -target="aws_route53_zone.main" -var-file="envs/prod.tfvars"

terraform output route53_nameservers
terraform output nameserver_setup_instructions
⚠ Atualize o registrador agora
Cole os 4 nameservers no provedor do domínio (veja a Seção 5). Aguarde as ferramentas públicas de DNS mostrarem os novos NS antes da Fase 2.

Fase 2 — apply completo

terraform plan  -var-file="envs/prod.tfvars"
terraform apply -var-file="envs/prod.tfvars"

O Terraform criará: hosted zone no Route 53 (se ainda não existir da Fase 1), certificado ACM (us-east-1), CNAMEs de validação DNS, registros alias A + AAAA, além de S3 + CloudFront. O apply completo leva cerca de 10–15 minutos após a propagação dos NS.

ℹ Espera da validação ACM
O Terraform fica bloqueado em aws_acm_certificate_validation até a AWS conseguir resolver o CNAME de validação via DNS público. Se der timeout, a propagação dos nameservers ainda não terminou — aguarde e execute terraform apply novamente.

Enviar arquivos HTML

Bash / macOS / Linux

BUCKET=$(terraform output -raw content_bucket_name)

aws s3 cp index.html s3://$BUCKET/index.html
aws s3 cp error.html s3://$BUCKET/error.html

PowerShell (Windows)

$BUCKET = terraform output -raw content_bucket_name

aws s3 cp index.html "s3://$BUCKET/index.html"
aws s3 cp error.html "s3://$BUCKET/error.html"

Conferir os outputs

Após um apply bem-sucedido, terraform output exibe:

apex_url = "https://ataqtech.com"
www_url = "https://www.ataqtech.com"
cloudfront_cdn_endpoint= "d1234abcd.cloudfront.net"
certificate_arn = "arn:aws:acm:us-east-1:..."
content_bucket_name = "landing-page-prod-static-website-content"
route53_zone_id = "Z0123456789ABC"
route53_nameservers = [
  "ns-8.awsdns-47.net",
  "ns-1.awsdns-60.org",
  "ns-1.awsdns-57.co.uk",
  "ns-1.awsdns-19.com",
]
nameserver_setup_instructions = "<guia passo a passo do registrador>"

5Configurar Nameservers no Registrador

O Terraform cria a hosted zone no Route 53, mas apontar o domínio para esses nameservers é um passo manual único no painel do registrador do domínio.

Onde encontrar os valores de nameserver

Após a Fase 1 (ou qualquer apply que tenha criado a zone), execute:

terraform output route53_nameservers
terraform output nameserver_setup_instructions

Ou localize-os no Console AWS → Route 53 → Hosted Zones → seu domínio → registro NS.

Exemplo de nameservers (os seus serão diferentes):

ns-8.awsdns-47.net
ns-1.awsdns-60.org
ns-1.awsdns-57.co.uk
ns-1.awsdns-19.com
⚠ Remova o ponto final
Alguns registradores não aceitam o ponto final (.). Remova-o se o formulário rejeitar os valores.

Passo a passo por registrador

RegistradorCaminho até as configurações de Nameserver
GoDaddyMy Products → DNS → Nameservers → Change → Enter my own nameservers
NamecheapDomain List → Manage → Nameservers → Custom DNS
Registro.brDomínios → seu domínio → Servidores DNS → Configurar
Cloudflare RegistrarDomain Registration → your domain → Nameservers
Google / SquarespaceDNS → Custom name servers → Use custom name servers
AWS Route 53 RegistrarNão é necessário — a hosted zone já é autoritativa

Verificar a propagação

A propagação leva de 5 minutos a 48 horas. Confira com uma ferramenta pública de DNS:

  • dnschecker.org — pesquise seu domínio e selecione o tipo de registro NS
  • whatsmydns.net — mostra resultados por região no mundo

Quando todas as regiões exibirem os valores NS do Route 53, a validação do ACM será concluída automaticamente no próximo terraform apply (ou já terá sido concluída se a propagação ocorreu durante o apply).

7Custo Estimado

As estimativas abaixo são calculadas com base nos preços de tabela da AWS para o ano corrente (2026) em us-east-1, usando preço pay-as-you-go e PriceClass_100. Os valores reais variam conforme tráfego, armazenamento e região. O registro do domínio (taxa do registrador) é cobrado à parte e não está incluído.

ℹ Base de preços
Os valores refletem o preço público da AWS em 2026, na época em que este guia foi escrito. Confira novamente as páginas de Pricing da AWS antes de orçar produção — as tarifas podem mudar.

Custo mensal por serviço (landing page de baixo tráfego)

Serviço dev prod Observações
Amazon S3 ~$0.01 ~$0.01 – $0.05 Armazenamento de HTML/assets (<1 GB). Transferência para o CloudFront é gratuita.
Amazon CloudFront ~$0 – $0.50 ~$0.10 – $1.50 Transferência de saída + requisições HTTPS. Com tráfego muito baixo, costuma ficar perto de $0; Free Tier pode valer para contas novas.
AWS Certificate Manager N/A Grátis Certificados públicos usados com CloudFront são gratuitos.
Amazon Route 53 N/A ~$0.50 Taxa da hosted zone ($0.50/mês). Consultas alias para CloudFront não são cobradas.
Total estimado / mês ~$0 – $0.50 ~$0.50 – $2 Landing page pequena típica (<10 GB de transferência / mês).

O que aumenta o custo

  • Transferência de dados do CloudFront — principal custo variável conforme o tráfego cresce (~$0.085/GB nas regiões PriceClass_100 após o Free Tier).
  • Invalidações de cache — os primeiros 1.000 paths/mês são gratuitos; depois, $0.005 por 1.000 paths.
  • Hosted zone do Route 53 — $0.50/mês fixos sempre que um domínio personalizado é usado (prod).
✓ Resumo de custo
Neste stack, o custo de infraestrutura costuma ficar abaixo de $2/mês para uma landing page de baixo tráfego. Em prod, a base é a hosted zone do Route 53; o restante escala com o número de visitantes.

8Invalidação de Cache

Após enviar novos arquivos HTML, o CloudFront pode servir a versão antiga em cache por até 24 horas (TTL padrão). Execute uma invalidação de cache para forçar a atualização imediata:

Bash / macOS / Linux

DIST_ID=$(terraform output -raw cloudfront_distribution_id)

# Invalida todos os arquivos
aws cloudfront create-invalidation \
  --distribution-id $DIST_ID \
  --paths "/*"

# Invalida apenas um arquivo específico
aws cloudfront create-invalidation \
  --distribution-id $DIST_ID \
  --paths "/index.html"

PowerShell (Windows)

$DIST_ID = terraform output -raw cloudfront_distribution_id

# Invalida todos os arquivos
aws cloudfront create-invalidation `
  --distribution-id $DIST_ID `
  --paths "/*"

# Invalida apenas um arquivo específico
aws cloudfront create-invalidation `
  --distribution-id $DIST_ID `
  --paths "/index.html"
ℹ Nota de custo
Os primeiros 1.000 paths de invalidação por mês são gratuitos. Cada 1.000 paths adicionais custam $0.005. Em dev, defina cloudfront_default_ttl = 0 no dev.tfvars para desativar o cache por completo.

9Destruir a Infraestrutura

⛔ Aviso — irreversível
Isso exclui permanentemente o bucket S3 de conteúdo e seus arquivos, a distribuição CloudFront, o certificado ACM, os registros do Route 53 e a hosted zone. Em prod, force_destroy = false faz o S3 recusar a exclusão se o bucket não estiver vazio — esvazie-o antes. Aponte o registrador para outros NS antes de destruir se ainda precisar do domínio em outro lugar.

Dev

terraform destroy -var-file="envs/dev.tfvars"

Prod

Bash / macOS / Linux

# Esvazie o bucket primeiro (prod tem force_destroy = false)
BUCKET=$(terraform output -raw content_bucket_name)
aws s3 rm s3://$BUCKET --recursive

# Em seguida, destrua
terraform destroy -var-file="envs/prod.tfvars"

PowerShell (Windows)

# Esvazie o bucket primeiro (prod tem force_destroy = false)
$BUCKET = terraform output -raw content_bucket_name
aws s3 rm "s3://$BUCKET" --recursive

# Em seguida, destrua
terraform destroy -var-file="envs/prod.tfvars"
